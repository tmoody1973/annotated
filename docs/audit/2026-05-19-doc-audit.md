## Dimension 3: Cross-Doc Contradictions

### [WARNING] Tailwind CSS v4 installation path is ambiguous
**Docs:** CLAUDE.md stack table ("Tailwind CSS v4 | install via @heroui/styles"), SETUP.md scaffold step 3 (`create-next-app@latest ... --tailwind`)
**Issue:** CLAUDE.md says Tailwind v4 is required and installed via `@heroui/styles`. SETUP.md's scaffold uses `create-next-app --tailwind`, which installs Tailwind independently (version determined by create-next-app defaults at scaffold time). The docs don't explain how these two installation paths interact, whether one overrides the other, or how to verify the correct version is active after setup. A developer following SETUP.md could end up with the wrong Tailwind version or a config conflict.
**Fix:** Add a note to SETUP.md step 9 (HeroUI install) clarifying that `@heroui/styles` manages the Tailwind config and the `--tailwind` flag from create-next-app provides the initial setup. If @heroui/styles requires a specific Tailwind version, call it out explicitly.

No other cross-doc contradictions found. All env var names, service names, and build order descriptions are consistent across docs.

### [WARNING] 90-second clip enforcement mechanism is not described
**Docs:** ARCHITECTURE.md podcast pipeline steps 7–8; YouTube pipeline steps 5–6
**Issue:** The 90-second cap is a hard spec requirement. ARCHITECTURE.md's ffmpeg commands use `{duration}` as a passthrough from `clipStartMs`/`clipEndMs` — but nowhere specifies who enforces the cap: the sidebar UI (prevent dragging past 90s), the Convex mutation (reject clips over 90s), or the worker (truncate). Without a clear enforcement point, a clip over 90 seconds could be submitted and processed. The spec says 90 seconds; the architecture doesn't say how.
**Fix:** Add one sentence to ARCHITECTURE.md step 7 (or the worker section) specifying where the 90-second validation lives. Recommended: enforce in the sidebar UI (disable drag past 90s boundary) AND validate server-side in the Convex mutation before calling the worker.

### [WARNING] Email delivery mechanism for claims is not specified
**Docs:** CLAUDE.md ("claims write to the DB and email Tarik"), SETUP.md step 10 ("File a claim form + email notification"), ARCHITECTURE.md (claims table, no email mention)
**Issue:** The spec requires a "File a claim" button. The docs say claims write to the DB and email Tarik, but no email service is listed in SETUP.md's service provisioning table, CLAUDE.md's stack table, or ARCHITECTURE.md. A developer implementing step 10 has no guidance on which email service to use (Resend, Postmark, SendGrid, Convex email action, etc.).
**Fix:** Add the email service to SETUP.md's service table and CLAUDE.md's stack table. Recommended: Resend (simplest setup, Convex-compatible). Add a note to ARCHITECTURE.md's claim handling describing the trigger (Convex scheduled function or action on claims insert → Resend API call).

### [MINOR] Source attribution requirement is implied, not explicit
**Docs:** CLAUDE.md ("source-linked landing page"), BUILD-INTENT.md — no doc explicitly calls out "every clip must link to its original source URL" as a hard requirement with a required field in the data model.
**Fix:** Add one sentence to ARCHITECTURE.md under the annotation landing page description confirming that the source URL is always visible and linked. The schema already has `canonicalUrl` on the sources table — this is mainly a documentation clarity issue.
